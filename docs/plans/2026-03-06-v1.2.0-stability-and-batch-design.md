## 背景

実機テスト（カタログ 32,000枚+）で以下の4つの問題を発見:

1. **Lua プラグイン安定性**: 検索系コマンドが大規模カタログでタイムアウト/クラッシュし、後続コマンドをブロック
2. **検索フィルタ不具合**: `catalog.findPhotos` のフィルタパラメータが全て無視され全件返却
3. **バッチ現像処理**: `LrDevelopController` がアクティブ写真にしか効かず、`batchApplySettings` が壊れている
4. **バージョンハードコード**: Lua 側 `version = "1.0.0"` が固定、Python 側 `__version__ = "1.1.0"` と不整合

## 問題1: Lua プラグイン安定性（大規模カタログ対応）

### 根本原因分析

- `catalog:getAllPhotos()` が 32,000枚全件をメモリに載せ、1回の `withReadAccessDo` 内でメタデータ取得を試みる
- `getCollections` で `#collection:getPhotos()` を全コレクションに対して呼び、写真数カウントが O(N*M) になる
- コマンドが `LrTasks.startAsyncTask` 内で同期的に実行され、1コマンドがソケットを占有する
- Lightroom の Lua sandbox には `withReadAccessDo` の実行時間制限があり、長時間ロックでクラッシュする

### アプローチ比較

| アプローチ | 説明 | 利点 | 欠点 |
|---|---|---|---|
| **A. チャンク分割処理** | `withReadAccessDo` を小さな単位（500枚ずつ）で分割し、各チャンク間で `LrTasks.yield()` を挟む | ロック時間短縮、他コマンド処理可能 | 実装が複雑、チャンク間でカタログ状態が変わる可能性 |
| **B. サーバーサイドページネーション** | Python 側でページサイズを管理し、1リクエストで取得する件数を制限。Lua 側は1ページ分だけ処理 | シンプル、既存APIに `offset`/`limit` あり | クライアントが複数回呼び出す必要あり、全件取得が遅い |
| **C. 遅延メタデータ取得** | ID リストだけ先に返し、メタデータは別コマンドで個別/バッチ取得 | 最速レスポンス、メモリ効率最良 | API の使い方が変わる、エージェントの呼び出し回数増加 |

### 推奨: A + B のハイブリッド（+ NDJSON ストリーミング）

**理由:**
- B のページネーションを基本とし、1リクエストあたりの処理量をデフォルト500枚に制限
- A のチャンク分割を Lua 内部で適用し、500枚の処理も50枚ずつの `withReadAccessDo` に分割
- 各チャンク間で `LrTasks.yield()` を呼び、他コマンド（ping 等）が処理可能になる
- **長期的には NDJSON ストリーミング（問題5参照）により、チャンクごとの逐次送信で根本解決**

**`getAllPhotos()` のメモリ負荷について（I1 対応）:**
- 短期的には `catalog:getAllPhotos()` による ID 配列の一括取得は許容する。ID のみであればメモリ負荷は軽い（32,000枚 x 8byte = 約256KB）
- メタデータ取得のみをチャンク分割することで、実質的なメモリ問題を回避する
- 長期的には NDJSON ストリーミング導入により、ID 取得もストリーミング化する

### 詳細設計

#### 1-1. CatalogModule の改修

**`findPhotos` / `getAllPhotos`:**

```
処理フロー:
1. catalog:getAllPhotos() で ID 配列を取得（メタデータなし、高速）
2. フィルタ条件でスキャン（チャンク50枚ずつ、withReadAccessDo を分割）
3. offset/limit で結果をスライス
4. メタデータ取得もチャンク50枚ずつ分割
5. 各チャンク間で LrTasks.yield()
6. NDJSON モード有効時: チャンクごとに sender 経由で data イベントを送信（問題5参照）
```

**チャンクサイズ設定:**
- `FILTER_CHUNK_SIZE = 50` — フィルタリングの1回あたりの処理枚数
- `METADATA_CHUNK_SIZE = 50` — メタデータ取得の1回あたりの処理枚数
- `DEFAULT_PAGE_SIZE = 500` — デフォルトの `limit` 値（現在の100から増加、ただしチャンク分割するため安全）
- `MAX_PAGE_SIZE = 2000` — 最大許容 `limit` 値

**`getCollections` の最適化:**
- `#collection:getPhotos()` を廃止し、`collection:getPhotoCount()` に置換（存在する場合）
- 存在しない場合は `photoCount` フィールドを省略（遅延取得に変更）
- コレクションセット内のネスト走査にも `yield()` を挿入

#### 1-2. コマンドタイムアウトの見直し

**Python 側 `retry.py`:**

```
変更:
- catalog.findPhotos: 30s -> 90s
- catalog.getAllPhotos: 60s -> 90s
- catalog.getCollections: 30s -> 60s
- 新規追加: catalog.getCollections のタイムアウトエントリ
```

**注:** `catalog.searchPhotos` は `findPhotos` に委譲されるため、個別のタイムアウト設定は `findPhotos` と同値にする。

#### 1-3. コマンドキャンセレーション（C1 対応: デュアルソケット活用）

**課題:** 現在、タイムアウトしたコマンドの処理が Lua 側で走り続ける。

**旧設計の矛盾（C1 指摘）:**
旧設計では「同じソケットで `cancelCommand` を後送する」としていたが、1コマンドがソケットを占有する状況ではキャンセル要求の受信もブロックされる。これは設計上の矛盾である。

**新設計: Lua 側タイムアウト自己検知方式**

デュアルソケットアーキテクチャ（sender: LR→Python, receiver: Python→LR）を前提とし、以下の方式でキャンセルを実現する:

1. **Lua 側のチャンク処理内タイムアウト自己検知:**
   - `CommandRouter` にコマンド開始時刻 `commandStartTime[requestId]` を記録
   - 各チャンク処理ループの先頭で経過時間をチェック
   - 経過時間がコマンド種別ごとの `LUA_COMMAND_TIMEOUT` を超えた場合、自発的に中断
   - 中断時は処理済みチャンクの部分結果 + `incomplete: true` + `reason: "timeout"` を返却

2. **Python 側のタイムアウト処理:**
   - Python 側でタイムアウトが発生した場合、`cancelledRequests` テーブルに `requestId` を追加する要求を **receiver チャネル（Python→LR）** 経由で送信
   - Lua 側の受信ループ（receiver ソケット処理）は別の `LrTasks.startAsyncTask` で動いているため、メインの処理ループとは独立してキャンセル要求を受信可能
   - チャンク処理ループで `cancelledRequests[requestId]` もチェックし、true なら中断

3. **二重の安全策:**
   - Lua 側自己検知（主）: ソケット通信に依存しない確実な中断
   - Python→Lua キャンセル通知（副）: receiver チャネル経由でより迅速なキャンセルを実現

```
Lua 側のタイムアウト設定:
- LUA_COMMAND_TIMEOUT (デフォルト): 80s（Python 側 90s より短く設定）
- LUA_COMMAND_TIMEOUT (preview系): 110s（Python 側 120s より短く設定）
- マージン 10s は、部分結果の返却とソケット通信に必要な時間
```

### エラーハンドリング（C2 対応: 部分成功の整合性保証）

- チャンク処理中にエラーが発生した場合、そのチャンクをスキップし次のチャンクを処理（部分的成功を許容）
- **`incomplete: true` フラグの強制付与**: チャンクのスキップが1件でも発生した場合、レスポンスに必ず `incomplete: true` を付与する。エージェントはこのフラグにより結果が不完全であることを検知可能
- レスポンスに `partialErrors` フィールドを追加し、スキップした写真の ID とエラー内容を返却
- `processedCount` / `totalCount` フィールドを追加し、処理済み/全体の件数を明示
- 全チャンクが失敗した場合はエラーレスポンスを返す

**レスポンス例（部分成功時）:**
```json
{
  "success": true,
  "incomplete": true,
  "reason": "chunk_errors",
  "photos": [...],
  "totalCount": 500,
  "processedCount": 480,
  "partialErrors": [
    {"photoId": 123, "error": "metadata_read_failed"},
    {"photoId": 456, "error": "metadata_read_failed"}
  ]
}
```

## 問題2: 検索フィルタの修正

### 根本原因分析

コードを精査した結果、2つの独立した問題が存在:

1. **`catalog.searchPhotos`**: フィルタリングロジックが存在しない。`criteria` パラメータを受け取るが、実際には `catalog:getTargetPhotos()` または `catalog:getAllPhotos()` の結果をそのままページネーションして返すだけ（L86-L189）
2. **`catalog.findPhotos`**: フィルタリングロジック自体は正しく実装されている（L586-L630）が、CLI/Python 側から `searchDesc` パラメータが正しく渡されていない可能性がある

### アプローチ比較

| アプローチ | 説明 | 利点 | 欠点 |
|---|---|---|---|
| **A. searchPhotos にフィルタ実装** | `searchPhotos` に `criteria` ベースのフィルタリングを追加 | 既存APIとの互換性維持 | `findPhotos` と機能重複 |
| **B. findPhotos に統一** | `searchPhotos` を deprecated にし、`findPhotos` にフィルタ機能を集約 | 単一API、メンテナンス容易 | 既存クライアントに影響 |
| **C. 両方を修正して棲み分け** | `searchPhotos` = 選択写真/全写真から検索、`findPhotos` = カタログ全体からフィルタ検索 | 用途が明確 | 2つのAPI を維持するコスト |

### 推奨: B（findPhotos に統一）

**理由:**
1. `searchPhotos` と `findPhotos` の機能が重複しており、エージェントが迷う原因になる
2. `findPhotos` には既にフィルタロジックが実装されている（ただしバグあり — 後述）
3. `searchPhotos` は `catalog:getTargetPhotos()` を使うが、これは「UI で現在表示されている写真」であり、エージェントからの操作では不確定

### 詳細設計

#### 2-1. findPhotos のフィルタ拡張

**現在サポートされているフィルタ:**
- `flag` (pick/reject/none)
- `rating` + `ratingOp` (==, >=, <=, >, <)
- `colorLabel`
- `camera`

**追加するフィルタ:**

| フィルタ | パラメータ名 | 型 | 説明 |
|---|---|---|---|
| フォルダパス | `folderPath` | string | 部分一致（`string.find` で照合） |
| 撮影日範囲（開始） | `captureDateFrom` | string (ISO 8601) | `dateTimeOriginal` 以降の写真を取得 |
| 撮影日範囲（終了） | `captureDateTo` | string (ISO 8601) | `dateTimeOriginal` 以前の写真を取得 |
| ファイル形式 | `fileFormat` | string | RAW / DNG / JPEG 等の完全一致 |
| キーワード | `keyword` | string | 写真に付与されたキーワード名の部分一致 |
| コレクション | `collectionName` | string | 所属コレクション名の部分一致 |
| ファイル名 | `filename` | string | 部分一致 |

**フィルタ実行の最適化:**
- 「軽い」フィルタ（rating, flag, colorLabel, fileFormat）を先に適用
- 「重い」フィルタ（keyword, collection, folderPath）は軽いフィルタで絞った後に適用
- チャンク分割（問題1の設計と連携）

#### 2-2. searchPhotos の廃止計画（C3 対応: 互換管理の明確化）

**背景:** このCLIは v1.1.0 が初の MCP/エージェント公開であり、`searchPhotos` の外部利用者は事実上いない。そのため、過度な互換期間は不要である。

- **v1.2.0:**
  - `searchPhotos` を内部で `findPhotos` に委譲
  - Lua 側のログに `deprecated: searchPhotos は findPhotos に統合されました。次バージョンで削除予定。` を出力
  - SKILL.md から `searchPhotos` の記述を **削除** し、`findPhotos` に統一
  - Python SDK の `searchPhotos` メソッドは残すが、docstring に deprecated を明記
- **v1.3.0:**
  - `searchPhotos` を Lua/Python 両側から完全削除

**互換期間が短い理由の根拠:**
- v1.1.0 以前は内部テストのみで外部利用者なし
- MCP/エージェント公開後のユーザーは SKILL.md を参照するため、SKILL.md を `findPhotos` に統一すれば実質的な影響なし

#### 2-3. findPhotos のバグ修正

現在の `findPhotos` にも以下のバグが存在:

1. **`pickStatus` フィルタのパラメータ名不一致**: CLI 側では `pickStatus` だが、Lua 側では `searchDesc.flag` を参照。Python SDK/CLI のパラメータマッピングを確認し統一する
2. **`captureDateFrom` / `captureDateTo` フィルタが未実装**: `searchDesc` に撮影日パラメータを渡しても対応するフィルタコードが存在しない
3. **フィルタ結果が0件の場合のレスポンス形式**: `callback` に `result` ラッパーが二重になる可能性

### エラーハンドリング

- 不明なフィルタキーが渡された場合: 無視せず、レスポンスの `warnings` フィールドに未サポートフィルタ名を列挙
- フィルタ値が不正な場合（例: rating に文字列）: エラーレスポンスを即座に返却
- フィルタ処理中のメタデータ取得失敗: その写真をスキップ（フィルタ不一致として扱う）

## 問題3: バッチ現像処理

### 根本原因分析

- `LrDevelopController.setValue()` は常に「現在アクティブな写真」に対して動作する
- `batchApplySettings` で `photoIds` をループしているが、各イテレーションで `catalog:setSelectedPhotos(photo, {photo})` を呼んでいないため、全設定が最初のアクティブ写真にだけ適用される
- `LrDevelopController` は Develop モジュールでのみ動作し、Library モジュールでは一部の操作が失敗する

### アプローチ比較

| アプローチ | 説明 | 利点 | 欠点 |
|---|---|---|---|
| **A. setSelectedPhotos + setValue** | ループ内で `catalog:setSelectedPhotos()` を呼んでアクティブ写真を切り替え、`LrDevelopController.setValue()` を適用 | 既存APIを活用、確実に動作 | UIが切り替わる（写真選択が変わる）、Develop モジュールにいる必要あり、速度が遅い |
| **B. photo:getDevelopSettings() / applyDevelopSettings()** | `LrPhoto` オブジェクトの `getDevelopSettings()` で設定をコピーし、`applyDevelopSettings()` で別写真に適用 | UI を変更しない、Library モジュールでも動作 | `applyDevelopSettings()` は存在するが、個別パラメータの変更には不向き（全設定の上書きになる） |
| **C. Develop Preset 経由** | 現像パラメータを一時的な Develop Preset として保存し、`photo:applyDevelopPreset()` で各写真に適用 | UI 非依存、LR のネイティブ機能 | Preset の作成/削除が必要、一時ファイル管理が複雑 |

### 推奨: A（setSelectedPhotos + setValue）+ 安全策

**理由:**
1. `LrDevelopController.setValue()` が最も信頼性が高く、個別パラメータの細かい制御が可能
2. B の `applyDevelopSettings()` は設定の完全上書きになるため、「Exposure だけ変更」のようなユースケースに不向き
3. C は Preset ファイルの管理が複雑で、エラー時の cleanup が困難
4. UI 切り替えの副作用は、処理完了後に元の選択状態を復元することで緩和可能

### 詳細設計

#### 3-1. batchApplySettings の修正

```
処理フロー:
1. 現在の選択状態を保存（catalog:getTargetPhotos(), catalog:getTargetPhoto()）
2. Develop モジュールであることを確認（LrApplicationView.getCurrentModuleName()）
3. 各 photoId についてループ:
   a. catalog:setSelectedPhotos(photo, {photo}) でアクティブ写真を切り替え
   b. LrTasks.sleep(0.1) で UI 更新を待機
   c. 各設定パラメータに LrDevelopController.setValue() を適用
   d. 結果を記録
4. 元の選択状態を復元
5. 結果を返却
```

#### 3-2. バッチサイズ制限

- 1回のバッチコマンドで処理する写真数を **最大50枚** に制限
- 50枚を超える場合はエラーを返し、クライアント側で分割呼び出しを要求
- 理由: `setSelectedPhotos` + `sleep(0.1)` で1枚あたり最低100ms、50枚で5秒。タイムアウト内に収まる

#### 3-3. 新規コマンド: develop.batchSetValue

既存の `batchApplySettings`（複数設定 x 複数写真）に加え、より単純な「1つの設定を複数写真に適用」するコマンドを追加:

```
コマンド: develop.batchSetValue
パラメータ:
  - photoIds: number[]  (最大50)
  - param: string       (設定名)
  - value: number       (設定値)
レスポンス:
  - results: [{photoId, success, error?}]
  - processed: number
  - succeeded: number
```

#### 3-4. Develop モジュール確認

- コマンド実行時に `LrApplicationView.getCurrentModuleName()` で現在のモジュールを確認
- Develop モジュール以外の場合:
  1. `LrApplicationView.switchToModule("develop")` で自動切替を試みる
  2. `LrTasks.sleep(0.5)` で切替完了を待機
  3. 切替失敗時はエラーレスポンスを返す（`MODULE_SWITCH_FAILED`）

#### 3-5. Python 側タイムアウト（I2 対応: 動的タイムアウト計算）

**旧設計:** 固定タイムアウト（120s）

**新設計:** 枚数に応じた動的タイムアウト計算

```
計算式:
  timeout = baseTimeout + perPhotoTimeout * photoCount

パラメータ:
  - baseTimeout = 10s（モジュール切替・選択状態保存/復元等の固定コスト）
  - perPhotoTimeout = 2s（1枚あたりのコスト: sleep(0.1) + 複数 setValue + マージン）
  - 最小タイムアウト = 30s
  - 最大タイムアウト = 120s

例:
  - 10枚: 10 + 2*10 = 30s
  - 50枚: 10 + 2*50 = 110s
  - 1枚: max(30, 10 + 2*1) = 30s
```

**理由（I2 指摘対応）:**
旧設計の `100ms * 枚数` の見積りは、`sleep(0.1)` のみを考慮しており、複数の `setValue` 呼び出し + Develop モジュールの内部処理 + UI 更新の遅延を無視していた。`perPhotoTimeout = 2s` は実測値ベースで余裕を持った設定とし、実機ベンチマークで微調整する。

**Python 側実装:**

```python
# retry.py に動的タイムアウト計算を追加
def calculate_batch_timeout(photo_count: int) -> float:
    BASE_TIMEOUT = 10.0
    PER_PHOTO_TIMEOUT = 2.0
    MIN_TIMEOUT = 30.0
    MAX_TIMEOUT = 120.0
    return min(MAX_TIMEOUT, max(MIN_TIMEOUT, BASE_TIMEOUT + PER_PHOTO_TIMEOUT * photo_count))
```

### エラーハンドリング

- **部分的成功**: 50枚中30枚が成功、20枚が失敗した場合でも、成功した30枚の結果は保持
- **選択状態の復元失敗**: ログに警告を出すが、コマンド自体は成功として扱う
- **Develop モジュール切替失敗**: 即座にエラーを返し、写真の処理は行わない
- **処理中のキャンセル**: 問題1で設計した Lua 側タイムアウト自己検知 + receiver チャネル経由のキャンセル通知を利用し、ループ内でチェック

### 副作用と制約事項

| 副作用 | 影響 | 緩和策 |
|---|---|---|
| UI の写真選択が変わる | ユーザーが手動操作中に混乱 | 処理完了後に元の選択を復元 |
| Develop モジュールへの自動切替 | Library での作業が中断 | 元のモジュールを記録し、処理後に復元を試みる |
| 処理速度（1枚あたり最大2s） | 大量写真で時間がかかる | バッチサイズ上限50枚、エージェントに分割呼び出しを推奨 |
| Undo 履歴が汚れる | 1枚ずつの undo エントリが作成される | `withWriteAccessDo` で1つの undo ブロックにまとめる |

## 問題4: プラグインバージョンの動的同期

### 根本原因分析

- `PluginInit.lua` L205: `version = "1.0.0"` がハードコードされている（`system.ping` レスポンス）
- `Info.lua` L33: `VERSION = { major=1, minor=0, revision=0, build=1 }` もハードコード
- Python 側 `lightroom_sdk/__init__.py`: `__version__ = "1.1.0"`
- 2箇所が独立してバージョンを管理しており、同期が取れていない

### アプローチ比較

| アプローチ | 説明 | 利点 | 欠点 |
|---|---|---|---|
| **A. ビルド時注入** | リリーススクリプトで `pyproject.toml` のバージョンを読み取り、Lua ファイルに書き込む | 確実、実行時コストゼロ | ビルドステップが必要、手動リリースで忘れるリスク |
| **B. Python → Lua 通信** | 接続確立時に Python 側からバージョンを送信し、Lua 側がレスポンスに使用 | 常に最新、ビルド不要 | 接続前の ping には使えない、通信の追加コスト |
| **C. 共有バージョンファイル** | `VERSION` テキストファイルを1つ置き、Python も Lua も読み取る | Single Source of Truth | Lua の sandbox 制約で `io.open` が使えない可能性 |

### 推奨: A（ビルド時注入）+ フォールバック

**理由:**
1. Lua sandbox では `io.open` や `os.getenv` が使えないため、C は不可能
2. B は接続確立前の状態（ping 応答等）でバージョンが不明になる
3. A のビルド時注入は確実で、GitHub Actions の CI/CD パイプラインに組み込める
4. 手動リリースのリスクは pre-commit hook + CI チェックで軽減

### 詳細設計

#### 4-1. バージョンの Single Source of Truth

- `pyproject.toml` の `version` フィールドを唯一の情報源とする
- 現在: `version = "1.1.0"` (pyproject.toml)

#### 4-2. ビルド時注入スクリプト

```
スクリプト: scripts/sync_version.py
処理:
1. pyproject.toml から version を読み取り
2. lightroom_sdk/__init__.py の __version__ を更新
3. lightroom_sdk/plugin/PluginInit.lua の version = "x.x.x" を更新
4. lightroom_sdk/plugin/Info.lua の VERSION テーブルを更新
```

#### 4-3. CI/CD 統合（I4 対応: CI での不整合検出）

- **GitHub Actions**: PyPI 公開ワークフロー（既存の `.github/workflows/publish.yml`）に `sync_version.py` を組み込む
- **pre-commit hook**: バージョン不整合を検出して警告（ブロックはしない — 開発中は不整合があっても良い）
- **CI チェック（新規）**: `scripts/check_version_sync.py` を CI パイプラインに追加し、バージョン不整合時にビルドを fail させる

```
スクリプト: scripts/check_version_sync.py
処理:
1. pyproject.toml の version を読み取り
2. lightroom_sdk/__init__.py の __version__ と比較
3. lightroom_sdk/plugin/PluginInit.lua の version と比較
4. lightroom_sdk/plugin/Info.lua の VERSION と比較
5. いずれかが不一致の場合、exit code 1 で終了 + 不一致箇所を表示

CI 統合:
- .github/workflows/ci.yml に check_version_sync ステップを追加
- PR マージ前に必ず実行し、不整合があればマージをブロック
```

- **リリースフロー**: `pyproject.toml` のバージョンを変更 → `sync_version.py` 実行 → コミット → タグ → リリース

#### 4-4. ping レスポンスの改善（I3 対応: version フィールド維持）

**旧設計（I3 指摘前）:**
```json
{ "pong": true, "timestamp": 1234567890, "pluginVersion": "1.2.0", "protocolVersion": "1" }
```

**新設計（I3 指摘反映）:**
```json
{ "pong": true, "timestamp": 1234567890, "version": "1.2.0", "protocolVersion": "1" }
```

- `version` フィールドは **リネームしない**。既存のまま維持し、正しいバージョン値（`sync_version.py` で注入された値）を返す
- `protocolVersion` を **追加フィールド** として足す（将来のプロトコル互換性チェック用）
- 破壊的変更を避け、既存クライアントとの互換性を維持する

## 問題5: NDJSON ストリーミング（新規）

### 背景

TASK.md で保留になっていた NDJSON（Newline Delimited JSON）ストリーミングを本設計に組み込む。これにより問題1の `getAllPhotos()` メモリ問題（I1）を長期的に根本解決し、大規模カタログでの安定性を向上させる。

### NDJSON の概要

- 1行1JSONオブジェクトで結果をストリーミング送信する形式
- 大規模カタログの検索結果を一括JSONではなくチャンクごとに逐次送信
- 各行は独立した JSON として解析可能

### 適用範囲

| コマンド種別 | 対象コマンド | NDJSON の用途 |
|---|---|---|
| 読み取り系 | `findPhotos`, `getAllPhotos`, `getCollections` | チャンク結果を `data` イベントとして逐次送信 |
| 書き込み系 | `batchApplySettings`, `batchSetValue` | 進捗イベント (`progress`) のみ NDJSON、最終結果は単一サマリ |

### 通信プロトコル設計

#### イベントタイプ

| タイプ | 用途 | 送信タイミング |
|---|---|---|
| `data` | チャンクデータ（写真メタデータ等） | チャンク処理完了ごと |
| `progress` | 進捗情報（処理済み件数/全体件数） | チャンク処理完了ごと |
| `final` | 最終サマリ（全体の統計情報） | 全チャンク完了時 |
| `error` | エラー情報 | エラー発生時 |

#### NDJSON メッセージフォーマット

各行は以下の形式の JSON オブジェクト:

```json
{"requestId": "abc-123", "type": "data", "chunkIndex": 0, "payload": {"photos": [...]}}
{"requestId": "abc-123", "type": "progress", "processed": 50, "total": 500}
{"requestId": "abc-123", "type": "data", "chunkIndex": 1, "payload": {"photos": [...]}}
{"requestId": "abc-123", "type": "progress", "processed": 100, "total": 500}
...
{"requestId": "abc-123", "type": "final", "payload": {"totalPhotos": 500, "processedCount": 500, "incomplete": false}}
```

**エラー時:**
```json
{"requestId": "abc-123", "type": "error", "error": "chunk_failed", "chunkIndex": 3, "details": "withReadAccessDo timeout"}
{"requestId": "abc-123", "type": "final", "payload": {"totalPhotos": 500, "processedCount": 450, "incomplete": true, "partialErrors": [...]}}
```

#### 通信チャネル

- **sender チャネル（LR→Python）** を利用してストリーミング送信
- 各メッセージに `requestId` を付与し、どのコマンドのレスポンスか識別
- 複数コマンドが並行して NDJSON ストリーミングを行う場合も `requestId` で分離可能

### チャンク分割との統合

```
処理フロー（Lua 側: findPhotos を例に）:
1. catalog:getAllPhotos() で ID 配列を取得
2. フィルタ条件でチャンク50枚ずつスキャン（withReadAccessDo を分割）
3. 各チャンク処理完了ごとに:
   a. sender チャネル経由で { type: "data", payload: { photos: [...] } } を送信
   b. sender チャネル経由で { type: "progress", processed: N, total: M } を送信
   c. LrTasks.yield() で他コマンドに制御を譲る
4. 全チャンク完了後に { type: "final", payload: { totalPhotos: M, ... } } を送信
```

**利点:**
- `withReadAccessDo` のロック時間がチャンク単位（50枚）に短縮される
- Python 側はチャンクを受信するたびにメモリに追記し、`final` 受信後にレスポンスを構築
- タイムアウトの回避: 各チャンクは短時間で完了するため、全体のタイムアウトリスクが低減

### Python SDK 側の設計

```
処理フロー（Python 側）:
1. コマンド送信（receiver チャネル経由）
2. sender チャネルの on_event() でストリーミングイベントを受信
3. requestId でイベントをフィルタリング
4. type == "data": チャンクデータを内部バッファに追記
5. type == "progress": 進捗コールバック呼び出し（オプション）
6. type == "final": 内部バッファの全データを集約してレスポンスを構築
7. type == "error": エラー情報を記録（final まで待ってから処理）
8. 集約済みレスポンスを呼び出し元に返却
```

**SDK API:**

```python
# client.py
async def find_photos(self, filters: dict, stream: bool = False) -> dict:
    """
    stream=False (デフォルト): 全チャンク受信後に集約された単一レスポンスを返す
    stream=True: AsyncGenerator[dict, None] を返す（将来拡張用）
    """
    pass
```

### CLI 側の設計

- CLI 側は既存の単一 JSON/text/table 出力を維持する
- Python SDK が NDJSON を集約して返すため、CLI は変更不要
- 将来的に `--stream` オプションを追加し、NDJSON をそのまま stdout に流すことも可能（v1.3.0 以降の検討事項）

### MCP Server 側の設計

- MCP Server も Python SDK 経由で NDJSON を集約して受信
- ツールレスポンスとしては集約済みの単一 JSON を返す
- MCP プロトコル自体にはストリーミング機能がないため、集約が必須

### 後方互換モード

- NDJSON ストリーミングはデフォルトで **有効**
- Python SDK の `stream` パラメータで制御（デフォルト `False` = 集約モード）
- Lua 側は常に NDJSON 形式で送信し、Python SDK が集約する
- 既存の単一JSON レスポンス形式は Python SDK/CLI/MCP Server から見て変更なし
